# 编码与测试规范

## 1. 开发规范

### 1.1 数据库操作
- **禁止使用 `SELECT *`**，必须显式列出所有查询字段
- 所有表必须有对应的实体类，字段使用 `@TableField(exist = false)` 标注非数据库字段
- Mapper XML 中的 resultMap 需显式映射每个字段

### 1.2 API设计
- RESTful风格，路径统一使用 `/api/v1/{module}/{resource}`
- 返回统一封装 `R<T>` 结构
- 分页返回 `IPage<T>`，包含 `records`, `total`, `size`, `current` 字段

### 1.3 前后端数据约定
- 后端返回的JSON字段必须与前端模板中使用的字段名完全一致
- 列表类数据返回空数组 `[]`，勿返回 `null`
- 状态类字段必须同时返回状态码和状态名称（如 `status` + `statusName`）

## 2. 调试与问题排查规范

### 2.1 问题定位流程（强制执行）

**当收到Bug报告时，必须按以下顺序排查：**

```
Step 1: 确认数据来源
         ↓
    检查API实际返回的JSON数据（查看浏览器Network面板或后端日志）
         ↓
    用户提供了什么数据？ ← 关键！不要忽视用户给的信息
         ↓
Step 2: 验证假设
         ↓
    如果假设是"后端问题" → 检查后端代码、SQL、日志
         ↓
    如果假设是"前端问题" → 检查前端代码、数据流、控制台
         ↓
Step 3: 验证修复
         ↓
    修复后必须验证：刷新页面、重启服务、重新测试
```

### 2.2 调试检查清单

**后端问题排查：**
- [ ] 检查编译是否通过（`mvn compile`）
- [ ] 检查SQL是否正确（查看MyBatis日志）
- [ ] 检查实体类字段是否与数据库表匹配
- [ ] 检查非数据库字段是否有 `@TableField(exist = false)`
- [ ] 检查返回数据是否包含前端需要的字段

**前端问题排查：**
- [ ] 检查浏览器控制台是否有报错
- [ ] 检查Network面板API响应数据
- [ ] 确认API返回的字段名与前端使用的一致
- [ ] 使用 `console.log()` 输出关键变量值
- [ ] 确认数据流：API返回 → 赋值 → 模板渲染

### 2.3 数据验证优先原则

**在修改代码之前，必须先确认数据是否正确：**

1. **查看实际API响应**：不要假设数据结构，要亲眼确认
2. **打印关键变量**：特别是用户提供的JSON/截图中的字段
3. **对比预期vs实际**：预期返回什么？实际返回什么？

**示例**：
```javascript
// 错误做法：假设数据存在
if (row.actualQty !== null) { // 假设actualQty存在

// 正确做法：先验证数据
if (row.actualQty != null) { // 使用 != 而非 !== 兼容undefined
// 或者更健壮：
if (row.actualQty !== undefined && row.actualQty !== null) {
```

### 2.4 JavaScript弱类型注意事项

| 表达式 | 结果 | 说明 |
|--------|------|------|
| `undefined === null` | `false` | 严格等于 |
| `undefined == null` | `true` | 宽松等于 |
| `null === null` | `true` | 严格等于 |
| `0 === null` | `false` | 数字不等于null |
| `'' === null` | `false` | 空字符串不等于null |

**前端判断规范**：
- 判断字段是否存在/已赋值：`field != null`（兼容 `undefined` 和 `null`）
- 判断字段为null：`field === null`
- 判断数组/对象非空：`Array.isArray(field) && field.length > 0`

## 3. 代码自测规范

### 3.1 提交前必须验证

**每次代码修改后，必须执行以下验证：**

1. **编译检查**：
   ```bash
   mvn compile  # 后端编译
   npm run lint  # 前端lint（如有）
   ```

2. **功能测试**（针对修改的功能）：
   - 启动后端服务
   - 启动前端服务
   - 测试修改相关的功能路径
   - 验证API返回数据正确

3. **回归测试**：
   - 测试与修改功能相关的其他功能
   - 确保没有引入新问题

### 3.2 测试数据准备

- 使用与生产环境相同结构的数据库
- 测试数据应覆盖：正常数据、边界数据、空数据
- 关键业务流程必须完整测试

### 3.3 日志规范

- 后端关键操作添加日志：`log.info()`, `log.error()`
- 前端关键变量使用 `console.log()` 输出
- 异常信息要包含足够的上下文（参数、状态、错误详情）

## 4. 常见问题速查

| 问题现象 | 可能原因 | 排查方向 |
|----------|----------|----------|
| 前端显示"未定义" | JSON字段名不一致 | 检查API响应和前端模板 |
| 列表为空 | 后端返回null而非[] | 检查Service返回值 |
| 按钮置灰不可点击 | 判断条件逻辑错误 | 检查 `!=` vs `!==` |
| 数据不刷新 | 缓存或状态未更新 | 刷新页面或重启服务 |
| SQL报错 | 表/字段不存在 | 检查Mapper XML和数据库 |
| 编译报错 | 类型不匹配/缺少import | 查看IDE提示或编译输出 |

---

## 5. 沟通规范

- 报告Bug时必须提供：API地址、请求方法、返回JSON（关键部分）
- 如果是页面问题，提供截图和浏览器控制台信息
- 修复后说明：问题原因、修复方案、验证方法
