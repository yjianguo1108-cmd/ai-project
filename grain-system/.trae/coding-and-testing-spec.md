# TRAE 项目编码与自测规范

 

## 1 用户级工作流程约定

此文件适用于所有 TRAE 项目，用于规范 AI 与开发者的协作流程。

***

### 1.1 任务管理

#### 1.1.1 会话开始流程

- 读取 `TRAE_TASK_QUEUE.md`，了解当前任务状态
- 读取项目概述与相关文档，确认上下文信息
- 若任务队列不存在，则自动创建并初始化

#### 1.1.2 任务状态定义

| **状态** | **格式示例**                     |
| ------ | ---------------------------- |
| 待确认    | `- [ ] 任务名称 (优先级, 工时) - 待确认` |
| 待处理    | `- [ ] 任务名称 (优先级, 工时)`       |
| 已完成    | `- [x] ✅ 任务名称 - YYYY-MM-DD`  |

***

### 1.2 标准任务流程

1. **记录**：将任务写入 `TRAE_TASK_QUEUE.md`，标记为「待确认」
2. **评估**：简单任务（<100 行）直接编码；复杂任务先进行预研与方案设计
3. **编码**：用户确认方案后再实现代码
4. **测试**：修复 Bug 必须先编写失败的测试用例，验证后标记为完成
5. **回归**：根据代码改动范围，确定对应的测试覆盖范围
6. **汇报**：汇总结果，列出所有修改文件与关键变更点

***

### 1.3 关键协作规则

- **测试优先**：修复 Bug 前，必须先编写能复现问题的单元测试（如 pytest）
- **数据安全**：删除 / 修改数据前必须先备份，并提供恢复脚本
- **代码质量**：禁止使用模拟数据、硬编码，遵循最小必要复杂度原则
- **开发/测试环境**：加字段的DDL成本极低，没有生产风险，保持代码与表结构一致是最佳实践。
- **并行处理**：涉及 3 个以上模块的复杂分析，使用多 Agent 并行处理
- **文档维护**：每次会话必须维护 `TRAE_TASK_QUEUE.md`，保持与项目进度同步

## 2 项目级编码规范

### 2.1 代码质量要求

1. **可读性优先**：变量 / 函数命名清晰，避免缩写，注释说明「为什么」而非「是什么」
2. **单一职责**：函数 / 方法不超过 50 行，类不超过 500 行
3. **防御性编程**：参数校验、异常处理、边界值覆盖完整
4. **无硬编码**：配置、常量、魔法值全部提取到配置文件或常量类
5. **依赖管理**：仅引入必要依赖，避免版本冲突，使用锁定文件（如`pom.xml`/`package.json`）

### 2.2 VO 与 Mapper 字段一致性检查清单

**修改任何 VO 或涉及列表/详情页字段回显时，必须执行以下检查**：

| # | 检查项                  | 操作内容                                              |
| - | -------------------- | ------------------------------------------------- |
| 1 | VO 文件 @JsonProperty  | 所有字段添加 `@JsonProperty`（驼峰命名与前端一致时）                |
| 2 | Mapper XML resultMap | 所有字段添加 `<result column="xxx" property="xxx"/>` 映射 |
| 3 | Mapper XML SELECT    | 查询所有需要的字段，包括计算字段（如 `CASE WHEN`、数学运算）              |
| 4 | convertToVO 基本字段     | 所有字段都调用 `vo.setXxx()`                             |
| 5 | convertToVO 关联字段     | 查询关联表获取名称/类型字段（如 farmerName, grainType）           |
| 6 | convertToVO 用户字段     | 通过 `userMapper` 查询用户真实姓名                          |
| 7 | 列表页 vs 详情页           | 确保两种查询方式返回的数据结构一致（列表用 Mapper XML，详情用 convertToVO） |

**典型遗漏场景**：

- 只改了 VO 文件的 `@JsonProperty`，没改 Mapper XML 的 resultMap
- 只改了 Mapper XML 的 SELECT，没改 resultMap 的映射
- 只在 convertToVO 中填充字段，没在 Mapper XML 中填充（导致列表页缺失）
- 只在 Mapper XML 中填充字段，没在 convertToVO 中填充（导致详情页缺失）

### 2.3 提交规范

- 每次提交必须关联任务编号，提交信息格式：`feat/fix/refactor/...: 描述 (#任务号)`
- 禁止提交临时调试代码、敏感信息，大体积文件
- 代码提交前必须通过本地编译与单元测试

## 3 自测与审计规范

### 3.1 适用场景

- ✅ 业务逻辑复杂的系统审计（如金融、交易、仓储系统）
- ✅ 关键基础设施审计
- ✅ 核心流程、数据流转正确性验证
- ❌ 纯技术性能优化
- ❌ 代码风格 / 格式问题

### 3.2 自测 - 审计 - 验证流程

#### 第一步：多模块并行审计

- 梳理系统业务架构，划分模块边界
- 对每个模块，从业务逻辑、数据流转、异常分支三个维度进行审计
- 记录所有潜在风险点与疑问

#### 第二步：审计结论质疑（辩护）

- 从业务逻辑角度对审计结论进行质疑，要求提供证据而非假设
- 典型效果：排除大量假阳性问题，聚焦真实风险

#### 第三步：真实场景测试验证

- 构造符合完整业务上下文的测试场景，而非孤立测试
- 测试用例必须包含：前置条件、执行步骤、预期结果
- 以可复现的测试结果作为最终验证依据

#### 第四步：动态风险评级

- 根据测试结果调整风险等级
- 避免过度反应，优先修复高风险、高影响的问题

### 3.3 自测核心原则

1. **业务逻辑 > 技术逻辑**
   - 必须先理解业务上下文、需求文档与设计方案
   - 阅读代码注释与设计文档，理解业务规则背后的含义
   - 禁止脱离业务逻辑进行纯技术层面的测试
2. **测试场景必须符合业务逻辑**
   - ❌ 错误：孤立测试单个变量 / 值（如 `assert value > threshold`）
   - ✅ 正确：覆盖完整业务场景，包含前置条件、数据流转与后置校验
3. **单元测试覆盖率要求**
   - 核心业务逻辑覆盖率≥90%
   - 异常分支、边界值必须覆盖
   - 禁止使用模拟数据跳过真实业务校验
