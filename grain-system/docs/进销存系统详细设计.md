**乡镇粮食代收点进销存系统概要设计文档**

**一、系统概述**

**1.1 系统定位**

本系统是面向乡镇粮食代收点的一体化进销存管理系统，聚焦粮食收购、仓储、销售全业务流程，整合用户管理、预约收购、称重计量、库存管控、订单销售、硬件数据对接等核心功能，实现乡镇粮食代收点业务的数字化、标准化、高效化管理，解决传统代收点人工记账效率低、数据易出错、库存管控不精准、业务追溯难等问题，适配基层操作人员的使用习惯，同时为代收点经营决策提供数据支撑。

**1.2 设计目标**

1. **业务全覆盖**：覆盖乡镇粮食代收点从农户预约、粮食收购、称重、入库、仓储盘点到下游销售、出库的全业务链路，实现进销存一体化管理；

2. **数据精准化**：建立统一的数据管理体系，实现各环节数据实时同步、自动流转，杜绝人工录入误差，保障账实相符、数据一致；

3. **操作便捷化**：简化操作流程，减少重复录入，适配乡镇基层操作人员的计算机操作水平，降低系统学习和使用成本；

4. **管理可视化**：通过数据统计、图表展示，实现库存状态、交易数据、业务进度的实时监控，为管理人员提供直观的经营视角；

5. **数据安全性**：建立完善的权限控制、操作留痕、数据加密机制，保障用户信息、交易数据、库存数据的安全，满足数据追溯要求；

6. **系统可扩展**：采用模块化设计，预留硬件对接、功能拓展接口，适配代收点后续业务升级和智能化改造需求。

**1.3 设计范围**

本系统覆盖**基础管理模块、收购管理模块、仓储管理模块、销售管理模块、硬件数据对接模块、统计分析模块**六大核心模块，同时包含数据库设计、系统架构设计、界面设计、性能与安全设计等内容，实现从用户身份认证、业务操作到数据统计、系统维护的全流程设计，适配乡镇粮食代收点的日常经营管理需求。

**1.4 设计原则**

1. **实用性原则**：以乡镇粮食代收点实际业务需求为核心，剔除冗余功能，确保系统功能贴合实际操作场景；

2. **模块化原则**：按业务功能划分子模块，模块间松耦合、内聚性强，便于系统开发、维护和功能拓展；

3. **数据一致性原则**：建立统一的数据标准，实现各模块数据自动同步、联动更新，确保收购、仓储、销售数据高度一致；

4. **易用性原则**：采用简洁的界面布局、清晰的操作流程，支持一键查询、批量操作、数据导入导出，提升操作效率；

5. **安全性原则**：遵循最小权限原则，建立角色化权限控制体系，所有操作全程留痕，敏感数据加密存储；

6. **兼容性原则**：适配主流计算机操作系统和浏览器，支持与地磅、温湿度传感器等基层常用硬件设备对接；

7. **可维护性原则**：系统代码、数据库设计遵循统一规范，配备完善的设计文档和操作手册，便于后期维护和升级。

**二、系统整体架构设计**

**2.1 技术架构**

系统采用**B/S（浏览器/服务器）架构**，结合分层设计思想，分为表现层、业务逻辑层、数据访问层、数据层四层架构，各层级职责清晰、解耦设计，便于开发、维护和扩展，整体技术架构如下：

1. **表现层**：为用户提供可视化操作界面，基于主流前端技术开发，支持电脑端主流浏览器（Chrome、Edge、Firefox），实现界面展示、用户交互、操作指令传递等功能，适配乡镇基层操作人员的使用场景；

2. **业务逻辑层**：系统核心层，包含六大核心模块的业务规则实现、流程控制、跨模块数据联动、权限校验等功能，接收表现层的操作指令，处理后传递至数据访问层，同时将处理结果反馈至表现层；

3. **数据访问层**：负责与数据库和文件存储的交互，封装数据CRUD操作，遵循统一的数据库访问规范，实现结构化数据与MySQL数据库的交互、非结构化数据与MinIO文件存储的对接，为业务逻辑层提供数据支撑；

4. **数据层**：包含MySQL 8.0关系型数据库和MinIO分布式文件存储，其中MySQL用于存储用户信息、交易数据、库存数据等结构化数据，MinIO用于存储监控视频、业务单据、资质文件、盘点照片等非结构化数据，保障数据存储的规范和高效。

**2.1.1 具体技术选型**

**前端技术栈：**Vue 3 + Vite + Element Plus。Vue 3 Composition API 利于逻辑复用，Element Plus 提供成熟的表单、表格、弹窗组件库，组件高度契合进销存系统的表单密集型操作场景，降低基层操作人员的学习成本。状态管理采用 Pinia，HTTP 请求封装采用 Axios，路由管理使用 Vue Router 4。

**后端技术栈：**Spring Boot 3.2.x + Java 17。采用 MyBatis-Plus 作为 ORM 框架，在 MyBatis 灵活 SQL 可控的基础上，提供单表 CRUD 自动化、分页插件、逻辑删除等增强能力。项目结构按 controller / service / mapper / entity 四层划分，统一返回体封装（Result\<T>），全局异常处理（GlobalExceptionHandler）。

**缓存方案：**Redis 7.x。用于以下场景：（1）用户登录 Token 存储与失效控制（TTL = 8 小时，强制下线时主动删除）；（2）角色权限数据缓存（角色菜单树、按钮权限码集合），减少高频权限校验的数据库查询；（3）粮食种类、储位、角色等基础配置数据本地缓存（TTL = 30 分钟，数据变更时主动失效）；（4）地磅设备最新称重数据缓存，供前端实时展示（TTL = 60 秒）。

**身份认证与安全：**JWT（JSON Web Token）无状态认证，Token 存储于 Redis，支持强制下线和 Token 刷新。密码采用 BCrypt 加盐哈希存储，登录失败 5 次后账号锁定 30 分钟（锁定计数器存 Redis）。HTTPS 强制跳转，服务端 TLS 证书本地自签或 Let's Encrypt。敏感字段（身份证号、银行卡号）采用 AES-256-GCM 对称加密，密钥存储于服务端配置文件，不入库。

**硬件通信协议：**地磅设备优先支持两种接入方式：RS-232/RS-485 串口直连（适配传统地磅，使用 jSerialComm 库读取串口数据流）和 TCP Socket 长连接（适配网络型地磅，服务端作为 Socket Server 监听固定端口，基于 Netty 框架实现高并发接入）。温湿度传感器和粮情监测设备推荐采用 MQTT 协议（Broker 使用 EMQX，服务端通过 Spring Integration MQTT 订阅设备上报 Topic）。监控摄像头视频流采用 RTSP 协议接入，前端通过 WebRTC 或 HLS 转码后播放。

**文件存储：**MinIO 单机部署（可平滑升级为分布式）。后端使用官方 MinIO Java SDK 进行文件上传、下载、预签名 URL 生成。文件访问采用预签名 URL（有效期可配置，默认 2 小时），避免直接暴露 MinIO 内网地址。

**构建与部署：**前端 Vite 构建为静态资源，由 Nginx 托管并配置反向代理转发 API 请求至后端。后端打包为可执行 JAR，使用 systemd 管理进程。数据库、MinIO、Redis、后端服务均部署于同一台本地服务器，Nginx 统一对外暴露端口（80/443）。

**2.2 系统部署架构**

系统采用**单机部署**模式（适配乡镇粮食代收点的经营规模和运维能力），部署于乡镇代收点本地服务器，包含应用服务器、数据库服务器、文件存储服务器，三者一体化部署，简化运维流程，具体部署架构如下：

• 应用服务器：部署系统前端和后端应用程序，接收用户操作请求，处理业务逻辑；

• 数据库服务器：部署MySQL 8.0数据库，存储系统所有结构化数据，配置定期自动备份策略；

• 文件存储服务器：部署MinIO分布式文件存储，存储系统非结构化数据，与数据库建立文件路径关联。

**三、核心模块详细设计**

**3.1 基础管理模块**

**3.1.1 核心子功能**

1. **用户管理**

￮ 实现管理员、操作员、农户、下游机构四类用户的新增、编辑、删除、状态管控（启用/禁用）；

￮ 支持用户信息查询、批量操作，记录用户创建时间、更新时间，密码加密存储，手机号唯一校验；

￮ 关联角色表，为用户分配对应角色，实现角色与用户的绑定。

2. **角色权限管理**

￮ 实现角色的新增、编辑、删除，支持角色名称、角色描述的配置，预设管理员、操作员、农户、下游机构四类基础角色；

￮ 基于RBAC权限模型，为不同角色分配不同的系统操作权限，细化至菜单权限、按钮权限，遵循最小权限原则；

￮ 支持角色权限的批量配置、复制，简化权限管理操作。

3. **农户档案管理**

￮ 实现农户档案的新增、编辑、查询、导出，关联用户表，同步农户基础信息；

￮ 加密存储农户身份证号、银行卡号，记录农户常用售粮种类，支持农户档案的批量导入/导出；

￮ 关联收购预约表、收购单据表，可查看农户历史售粮记录。

4. **下游机构档案管理**

￮ 实现下游机构档案的新增、编辑、查询、导出，关联用户表，同步机构基础信息；

￮ 存储机构资质文件路径（关联MinIO），实现机构资质的线上管理，支持资质文件的上传、查看、下载；

￮ 实现机构审核状态管控（待审核/审核通过/审核驳回），审核通过后方可进行销售订单操作；

￮ 关联销售订单表，可查看机构历史采购记录。

5. **基础配置管理**

￮ 实现粮食种类、粮食等级、储位、质量标准、仓储要求等基础数据的新增、编辑、删除、启用/禁用；

￮ 支持基础数据的批量导入/导出，为收购、仓储、销售模块提供统一的基础数据标准；

￮ 实现系统参数配置（如盘点损耗阈值、出库偏差阈值、预警规则），支持参数的灵活调整。

**3.1.2 业务规则**

1. 用户名、手机号、农户身份证号为唯一标识，不可重复；

2. 敏感信息（密码、身份证号、银行卡号）必须加密存储，不可明文展示；

3. 用户状态为禁用时，无法登录系统；下游机构审核状态为驳回/待审核时，无法创建销售订单；

4. 基础数据一旦被其他模块引用，不可删除，仅可标记为禁用；

5. 所有基础信息的修改、删除操作均记录操作日志，实现可追溯。

**3.1.3 状态流转设计**

**用户账号状态流转：**正常（active） → 禁用（disabled，操作员/管理员手动禁用，禁用后立即失效登录 Token）；禁用 → 正常（管理员手动启用）；正常 → 锁定（locked，登录失败 5 次后系统自动锁定，30 分钟后自动解锁或管理员手动解锁）。用户删除采用逻辑删除（is\_deleted = 1），数据保留用于历史追溯，不可物理删除。

**下游机构审核状态流转：**待审核（pending） → 审核通过（approved） → 审核驳回（rejected）；驳回后机构可重新提交资料，状态重置为待审核。审核通过后方可创建销售订单。机构账号禁用时，即使审核状态为通过，也无法操作销售业务。

**基础数据生命周期：**粮食种类、储位、质量标准等基础数据启用后可正常被其他模块引用；禁用后不再出现在下拉选项中，但已有历史业务数据中的引用保持不变。已被业务数据引用的基础数据不可物理删除，系统自动拦截并提示"该数据已被使用，无法删除"。

**3.1.4 核心接口设计（RESTful API）**

**用户管理接口：**

POST /api/v1/system/user — 新增用户（body：username、password、real\_name、phone、role\_ids\[]，密码后端 BCrypt 加密）

PUT /api/v1/system/user/{id} — 编辑用户信息（不含密码，密码单独接口修改）

PUT /api/v1/system/user/{id}/status — 启用/禁用用户（body：status=active/disabled，禁用时同步删除 Redis 中的 Token）

PUT /api/v1/system/user/{id}/password/reset — 管理员重置密码（生成随机密码，首次登录强制修改）

GET /api/v1/system/user — 分页查询用户列表（query：keyword、role\_id、status、page、size）

GET /api/v1/system/user/{id} — 查询用户详情（脱敏返回，手机号中间四位隐藏）

**角色权限接口：**

POST /api/v1/system/role — 新增角色

PUT /api/v1/system/role/{id}/permissions — 为角色批量分配权限（body：permission\_ids\[]，覆盖式更新，分配成功后刷新 Redis 中该角色的权限缓存）

GET /api/v1/system/role/{id}/permissions — 查询角色已有权限树（前端据此渲染权限配置界面）

GET /api/v1/auth/permissions — 当前登录用户的权限码集合（前端用于控制按钮显隐）

**农户档案接口：**

POST /api/v1/farmer/profile — 新增农户档案（body：user\_id、id\_card、bank\_card、bank\_name、grain\_types\[]，id\_card 和 bank\_card 后端 AES 加密存储）

GET /api/v1/farmer/profile/{id} — 查询农户详情（id\_card 和 bank\_card 脱敏展示，仅管理员可请求明文解密接口）

GET /api/v1/farmer/profile/{id}/decrypt — 查询农户敏感字段明文（需管理员权限，调用记录写操作日志）

GET /api/v1/farmer/profile/{id}/history — 查询农户历史售粮记录（分页，关联 t\_purchase\_order）

POST /api/v1/farmer/profile/import — 批量导入农户（multipart/form-data，Excel 文件，后端校验去重，返回成功/失败明细）

GET /api/v1/farmer/profile/export — 导出农户档案（Excel，敏感字段脱敏，仅管理员可用）

**下游机构接口：**

POST /api/v1/downstream/org — 新增下游机构（body：org\_name、contact\_name、phone、address，资质文件单独上传）

POST /api/v1/downstream/org/{id}/credential — 上传资质文件（multipart，存 MinIO 返回 file\_url，写入 t\_downstream\_org.credential\_url）

PUT /api/v1/downstream/org/{id}/audit — 机构审核（body：audit\_status=approved/rejected，reject\_reason）

GET /api/v1/downstream/org/{id}/orders — 查询机构历史采购记录（分页）

**基础配置接口：**

GET /api/v1/config/grain-types — 查询粮食种类列表（带等级嵌套，前端下拉选项用，Redis 缓存 30 分钟）

GET /api/v1/config/storage-positions — 查询储位列表（含当前库存量，供仓储模块选择储位用）

PUT /api/v1/config/system-params — 更新系统参数（body：key-value 键值对，如 price\_deviation\_max=0.2，修改后刷新相关缓存）

**3.1.5 关键数据表字段设计**

**t\_user（用户表）：**id INT PK AUTO\_INCREMENT; username VARCHAR(32) UNIQUE NOT NULL — 登录账号; password VARCHAR(128) NOT NULL — BCrypt 加密密码; real\_name VARCHAR(32) — 真实姓名; phone VARCHAR(20) UNIQUE — 手机号（登录备用）; status TINYINT DEFAULT 1 — 状态（0禁用/1正常/2锁定）; login\_fail\_count TINYINT DEFAULT 0 — 连续登录失败次数; lock\_time DATETIME — 锁定到期时间; last\_login\_time DATETIME — 最后登录时间; last\_login\_ip VARCHAR(64) — 最后登录IP; is\_deleted TINYINT DEFAULT 0 — 逻辑删除; create\_time DATETIME; update\_time DATETIME。

**t\_role（角色表）：**id INT PK AUTO\_INCREMENT; role\_name VARCHAR(32) UNIQUE NOT NULL; role\_code VARCHAR(32) UNIQUE NOT NULL — 角色编码（如 ADMIN、OPERATOR、FARMER）; description VARCHAR(200); is\_system TINYINT DEFAULT 0 — 是否系统内置角色（内置角色不可删除）; status TINYINT DEFAULT 1; create\_time DATETIME; update\_time DATETIME。

**3.1.6 关键业务规则补充与边界条件**

用户唯一性校验：username 和 phone 全局唯一，新增/编辑时后端实时校验，手机号格式校验采用正则（1\[3-9]\d{9}）。农户的 id\_card 字段在解密后比对唯一性（加密后密文不同明文可能相同），新增农户时需先解密现有数据进行去重校验，性能敏感时可存储 id\_card 的 SHA-256 哈希值（t\_farmer.id\_card\_hash）用于唯一索引。

权限缓存一致性：角色权限变更后，需立即刷新 Redis 中所有持有该角色的在线用户的权限缓存（通过 Redis Pub/Sub 广播失效消息）。用户被禁用时，除删除 Token 外，还需清空其权限缓存。

初始密码策略：新建用户初始密码由系统随机生成（8位，含大小写字母+数字+特殊字符），通过系统内消息通知操作员，首次登录强制修改密码（前端拦截跳转改密页面，后端校验 must\_change\_password 字段）。

操作日志规范：基础管理模块所有写操作（新增、编辑、删除、审核、状态变更）均写入 t\_operation\_log，字段：module=system、resource（user/role/farmer/org/grain/position）、operation（create/update/delete/audit/status\_change）、target\_id、before\_data（JSON）、after\_data（JSON）、operator\_id、operate\_time、ip\_addr。日志表不允许 DELETE 和 UPDATE 操作，仅允许 INSERT 和 SELECT。

批量导入容错处理：农户批量导入采用逐行校验策略，校验失败的行记录错误原因，其余行正常导入；返回体包含成功数、失败数、失败明细（行号+错误原因）供操作员下载对照。

**3.2 收购管理模块**

**3.2.1 核心子功能**

1. **收购预约管理**

￮ 支持农户端自主提交收购预约，填写预约粮食种类、预估重量、预约时间，系统自动生成预约单号；

￮ 支持操作员端对预约单的审核、取消、查询，可查看预约农户信息、预约详情；

￮ 支持预约单的批量处理，对已审核的预约单进行排期，方便代收点合理安排收购工作；

￮ 关联农户表，仅已建档的农户可提交预约，实现农户与预约单的一对一绑定。

2. **称重计量管理**

￮ 对接地磅设备表，实现地磅设备数据的自动采集，无需人工录入称重数据；

￮ 支持手动补录称重数据（针对设备故障场景），生成称重记录表，包含毛重、皮重、净重、称重时间、地磅设备ID、操作用户ID等信息；

￮ 实现称重记录的审核、作废，审核通过后的称重记录不可随意修改，关联收购单据表，为收购单据提供重量数据。

3. **收购单据管理**

￮ 支持基于收购预约单、称重记录表自动生成收购单据，自动回填农户信息、粮食种类、等级、重量、参考价格等数据；

￮ 支持操作员手动调整实际收购价格（不偏离参考价格过多），填写收购备注，上传收购单据附件（关联MinIO）；

￮ 实现收购单据的审核、作废、打印、导出，审核通过后触发仓储入库登记，同时生成农户付款待办；

￮ 关联农户表、粮食表、称重记录表，实现多表数据联动，一个收购单据对应一个称重记录。

4. **收购付款管理**

￮ 基于审核通过的收购单据生成付款单，自动计算付款金额（重量×实际收购价格）；

￮ 支持操作员标记付款状态（未付款/已付款/部分付款），填写付款时间、付款方式、付款流水号；

￮ 关联农户银行卡信息，实现付款信息与农户档案的联动，支持付款记录的查询、导出、打印；

￮ 付款完成后，系统自动更新收购单据的付款状态，记录付款操作日志。

**3.2.2 业务规则**

1. 收购预约单需先审核，审核通过后方可进行称重操作；

2. 称重记录表需先审核，审核通过后方可生成收购单据；

3. 收购单据审核通过后，不可随意修改，仅可作废后重新生成；

4. 收购付款金额需与收购单据的结算金额一致，部分付款需记录未付款金额，待后续补付；

5. 收购单据审核通过后，自动触发仓储管理模块的入库登记，实现收购与入库的无缝衔接；

6. 所有收购操作（预约、称重、制单、付款）均记录操作日志，关联操作用户ID。

**3.2.3 收购全流程状态机设计**

收购管理模块涉及预约、称重、单据、付款四个子流程，各子流程独立维护状态字段，通过业务联动实现全链路追溯。

**预约单（t\_purchase\_reserve.status）状态流转：**

待审核（pending）→ 已审核（approved）→ 已排期（scheduled）；驳回路径：待审核 → 已取消（cancelled）；操作员可主动取消已审核预约单。

**称重记录（t\_weighing\_record.status）状态流转：**

待审核（pending）→ 审核通过（confirmed）；作废路径：待审核 → 已作废（voided）。称重记录审核通过后不可修改，仅可作废后重新录入。地磅自动采集和操作员手动录入均走同一审核流程。

**收购单据（t\_purchase\_order.status）状态流转：**

草稿（draft）→ 待审核（pending）→ 审核通过（approved）→ 已完成（completed）；作废路径：审核通过 → 已作废（voided，需填写作废原因）。收购单据审核通过后自动触发两个动作：创建仓储入库单（status=待确认）、创建付款单（status=未付款）。

**付款单（t\_purchase\_payment.status）状态流转：**

未付款（unpaid）→ 部分付款（partial）→ 已付款（paid）。每次付款操作记录付款金额、付款方式、付款时间、操作员ID，累计付款金额达到应付金额时自动转为已付款状态。

**3.2.4 核心接口设计（RESTful API）**

后端统一前缀：/api/v1，使用 JWT Bearer Token 鉴权，响应体格式：{"code": 0, "msg": "success", "data": {...}}。

PUT /api/v1/purchase/reserve/{id}/audit — 操作员审核（body：status=approved/rejected，reject\_reason）

**称重记录接口：**

PUT /api/v1/purchase/weighing/{id}/audit — 审核称重记录（body：status=confirmed/voided）

**收购单据接口：**

PUT /api/v1/purchase/order/{id}/audit — 审核收购单据（审核通过后触发入库单和付款单的自动创建，采用数据库事务保障原子性）

GET /api/v1/purchase/order/{id} — 查看单据详情（含关联的称重记录、付款状态、入库状态）

**付款管理接口：**

POST /api/v1/purchase/payment/{id}/pay — 记录付款（body：pay\_amount、pay\_method、pay\_time、flow\_no）

GET /api/v1/purchase/payment — 查询付款列表（支持按付款状态、农户、时间段过滤、导出 Excel）

**3.2.6 关键业务规则补充与边界条件**

价格偏离控制：实际收购单价不得低于参考价格的 80%，不得高于参考价格的 120%，超出范围需填写说明并由管理员审核，阈值可在基础配置中调整。

重量偏差处理：实际收购重量与预约预估重量偏差超过 20% 时，系统提示操作员确认，偏差原因记录在备注字段。

事务一致性：收购单据审核通过触发入库单和付款单创建，三个操作在同一数据库事务中执行；任一失败则全部回滚，同时推送操作失败提醒。

数据锁定规则：称重记录一旦关联收购单据，不可重复关联；收购单据审核通过后，关联的称重记录状态同步更新为已使用，无法再被其他单据引用。

并发控制：同一地磅设备同一时刻仅允许一条称重记录处于待审核状态，防止重复提交。

操作权限细化：预约单的创建权限开放给农户角色；称重录入和单据创建权限分配给操作员角色；审核权限分配给管理员角色。所有操作写入 t\_operation\_log，字段包含：module（purchase）、operation（create/audit/void）、target\_id、before\_data、after\_data、operator\_id、operate\_time、ip\_addr。

**3.3 仓储管理模块**

**3.3.1 核心子功能**

1. **入库管理**

￮ 支持基于审核通过的收购单据自动生成入库单，自动回填粮食ID、种类、等级、收购重量、农户信息、操作用户等数据；

￮ 支持手动补录入库单（针对调拨入库、历史数据补录等场景），填写入库信息、选择储位、上传入库单据附件；

￮ 实现入库单的确认、作废、打印、导出，确认后系统自动更新库存表，实现库存数量的累加；

￮ 支持入库单的多条件查询（按入库时间、粮食种类、农户、操作用户），查看入库明细。

2. **库存管理**

￮ 实现库存数据的实时监控，支持多维度查询（按粮食种类、等级、储位、入库时间、库存数量），展示库存明细；

￮ 实现储位的精细化管理，支持储位的新增、编辑、删除、状态标记（可用/不可用），按储位记录库存数据；

￮ 实时展示库存总量、各类粮食库存占比，实现库存不足、库存积压、仓储超期等预警，预警信息在系统首页醒目展示；

￮ 生成库存台账，按日/周/月/年统计库存期初数、入库数、出库数、期末数，支持台账的查询、打印、导出。

3. **盘点管理**

￮ 支持创建定期/临时盘点计划，选择盘点范围（全量库存/指定粮食种类/指定储位），分配盘点操作员，设置盘点截止时间；

￮ 支持盘点操作员录入实际盘点数量，批量导入盘点数据，上传盘点现场照片/单据（关联MinIO）；

￮ 系统自动进行账实对比，生成盘点差异表，展示差异粮食信息、差异数量、差异率；

￮ 实现盘点差异的审核、处理，填写差异原因（损耗、录入错误、被盗等），选择处理方式（调账/重新盘点），审核通过后系统自动修正库存数据；

￮ 盘点完成后自动生成盘点报告，包含盘点范围、盘点人员、账实相符率、差异明细、处理结果，支持报告的归档、查看、打印。

4. **出库管理**

￮ 支持基于审核通过的销售订单自动生成出库单，自动回填粮食种类、等级、出库重量、下游机构信息、储位等数据；

￮ 支持手动录入出库单（针对调拨出库、应急出库等场景），填写出库信息、选择储位、上传出库单据附件；

￮ 系统自动校验库存充足性，实际出库重量不得大于账面库存重量（允许少量配置内偏差）；

￮ 实现出库单的确认、作废、打印、导出，确认后系统自动扣减对应储位的库存数量，更新库存台账；

￮ 支持出库单的多条件查询（按出库时间、粮食种类、下游机构、操作用户），查看出库明细。

**3.3.2 业务规则**

1. 入库单需关联唯一的收购单据或填写入库原因，无有效依据不得确认入库；

2. 储位标记为不可用后，该储位下的库存不可进行出库操作，需先完成库存调拨；

3. 盘点差异未处理前，对应库存不可进行出库操作，避免账实差异扩大；

4. 盘点损耗超出配置阈值时，需单独审核并详细记录原因；

5. 出库单需关联唯一的销售订单或填写出库原因，库存不足时禁止确认出库；

6. 入库、出库、盘点操作完成后，库存表实时更新，确保账实数据一致；

7. 所有仓储操作均记录操作日志，关联操作用户ID，日志不可删除。

**3.3.3 库存记录状态机设计**

t\_inventory 表中每条库存记录对应某一储位下某一粮食品种的实时库存快照，current\_stock 字段实时变动，同时通过 status 字段追踪库存的业务状态，共分为五态。

**待入库（pending\_in）：**收购单据审核通过后自动创建入库单，此时库存记录 status 置为待入库，current\_stock 暂不变更，防止入库单确认前数量被误用。

**在库（in\_stock）：**入库单操作员确认后 current\_stock 累加净重，status 转为在库，为正常可操作状态，允许出库和盘点。

**盘点锁定（checking）：**盘点计划发布且覆盖该储位时，对应库存记录 status 自动更新为盘点锁定。锁定期间禁止创建新的出库单，已审核但未执行的出库单暂停，防止账实差异在盘点过程中扩大。盘点结束（无论是否有差异）自动解锁。

**差异待处理（diff\_pending）：**盘点审核发现账实不符，对应库存 status 置为差异待处理。此状态下出库操作同样被禁止，直至差异审核处理完成（调账或重新盘点）。

**出库中（outbound）：**销售出库单确认操作期间的短暂过渡态，出库操作在数据库事务中执行，事务完成后自动扣减 current\_stock，status 恢复为在库（若库存仍有余量）或更新为已清零。

库存守恒公式（系统每日定时校验）：某储位当前库存 = 所有已确认入库净重之和 − 所有已确认出库净重之和 − 盘点负差异调账量 + 盘点正差异调账量。校验不通过时触发告警，写入 t\_operation\_log 并通知管理员排查。

**3.3.4 盘点闭环流程设计**

**阶段一：创建盘点计划（draft）：**管理员在系统创建盘点计划，选择盘点类型（全量/指定粮食品种/指定储位），设定截止时间，分配盘点操作员（可多人）。计划创建后系统自动为覆盖范围内的所有库存记录生成盘点明细行（t\_inventory\_check\_detail），预填账面数量（book\_qty），操作员实盘数量（actual\_qty）初始为空。

**阶段二：盘点进行中（in\_progress）：**计划发布后，覆盖库存记录自动锁定（status → checking）。操作员逐行录入实盘数量，支持批量 Excel 导入。每行录入后系统实时计算差异量（diff\_qty = actual\_qty − book\_qty）和差异率（diff\_rate = diff\_qty / book\_qty × 100%），超过损耗阈值（默认 2%，t\_system\_param 可配置）的行自动标红提示。操作员可上传现场照片（MinIO），作为差异证明。

**阶段三：待审核（pending\_audit）：**操作员提交盘点结果，系统汇总所有明细行生成盘点报告草稿：盘点范围、操作员、账实相符率、差异明细。管理员审核盘点报告，对差异明细逐行确认。

**阶段四A：无差异（completed）：**全部明细 diff\_qty = 0 或差异均在允许范围内，管理员确认通过，盘点完成。覆盖库存记录自动解锁（status → in\_stock），生成正式盘点报告归档。

**阶段四B：有差异处理（diff\_review → diff\_handling）：**存在超阈值差异，管理员标记有差异，进入差异处理阶段。处理方式二选一：① 调账——管理员填写差异原因（损耗/录入错误/被盗等），确认调账后系统自动修正 current\_stock（写入调账记录 t\_inventory\_adjustment，操作不可撤销，需二次确认），同时写操作日志；② 重新盘点——将盘点计划状态回退至 in\_progress，清空差异行的 actual\_qty，重新分配操作员执行实盘。处理完成后盘点转为 completed，库存解锁。

**盘点报告自动归档：**盘点完成后系统自动生成 PDF 盘点报告（含范围、人员、时间、差异明细、处理结果），存储于 MinIO /inventory/check-report/{yyyy-MM}/{check\_id}.pdf，关联记录在 t\_inventory\_check.report\_url，支持随时查看和重新打印。

**3.3.5 库存预警机制**

**库存不足预警：**当某储位某品种库存量低于 low\_stock\_threshold（t\_grain 表配置，默认 500 kg）时，系统首页展示预警提示，推送 WebSocket 通知在线管理员。预警解除条件：新增入库后库存量超过阈值，系统自动清除预警标记。

**库存积压预警：**某库存批次入库超过 high\_stock\_days（t\_system\_param 配置，默认 90 天）仍有余量，触发积压预警，提示经营者加快销售处理。

**仓储超期预警：**根据粮食品种的建议最长仓储时长（t\_grain.max\_storage\_days）计算每批库存的到期时间，提前 15 天触发预警。

**3.4 销售管理模块**

**3.4.1 核心子功能**

1. **销售订单管理**

￮ 支持操作员为审核通过的下游机构创建销售订单，选择粮食种类、等级、储位、销售数量，系统自动回填粮食参考价格；

￮ 支持手动调整实际销售价格，填写订单备注，上传订单附件（关联MinIO），系统自动生成销售订单号；

￮ 实现销售订单的审核、作废、修改、打印、导出，审核通过后触发仓储管理模块的出库登记；

￮ 支持销售订单的多条件查询（按订单时间、下游机构、粮食种类、订单状态），查看订单明细，跟踪订单进度。

2. **出库核销管理**

￮ 基于审核通过的销售订单生成出库待办，关联仓储管理模块的出库单，实现订单与出库单的一对一绑定；

￮ 支持操作员查看出库进度，确认实际出库数量，核对出库信息与订单信息的一致性；

￮ 出库完成后，系统自动更新销售订单的状态（待出库/已出库/部分出库），实现订单与出库的核销。

3. **销售收款管理**

￮ 基于已出库的销售订单生成收款单，自动计算收款金额（销售数量×实际销售价格）；

￮ 支持操作员标记收款状态（未收款/已收款/部分收款），填写收款时间、收款方式、收款流水号；

￮ 关联下游机构表，实现收款信息与机构档案的联动，支持收款记录的查询、导出、打印；

￮ 收款完成后，系统自动更新销售订单的收款状态，记录收款操作日志。

4. **客户对账管理**

￮ 支持按下游机构、时间区间生成对账报表，包含订单数量、出库数量、应收金额、已收金额、未收金额；

￮ 支持对账报表的打印、导出，支持与下游机构的线上对账、确认；

￮ 记录对账记录，关联对账人员、对账时间、对账结果，实现对账流程的可追溯。

**3.4.2 业务规则**

1. 仅审核通过的下游机构可创建销售订单；

2. 销售订单的粮食种类、等级、储位需与库存表数据一致，销售数量不得大于对应储位的库存数量；

3. 销售订单审核通过后，不可随意修改，仅可作废后重新生成；

4. 出库完成后，方可进行收款操作，收款金额需与订单结算金额一致，部分收款需记录未收款金额；

5. 销售订单的状态随出库、收款操作自动更新，实现订单全流程状态跟踪；

6. 所有销售操作（制单、出库、收款、对账）均记录操作日志，关联操作用户ID。

**3.4.3 三段状态联动设计**

销售管理模块的核心复杂度在于销售订单、出库核销、收款记录三张单据的状态相互联动。三段单据不独立推进，而是形成严格的先后依赖关系：订单审核通过才能执行出库，出库完成才能发起收款。任意一段的状态变更均需反写上游或联动下游，通过数据库事务保障原子性。

**销售订单（t\_sales\_order.status）七态流转：**草稿（draft）→ 待审核（pending）→ 待出库（approved）→ 部分出库（partial\_out）→ 已出库（out\_done）→ 已完成（completed）；驳回路径：待审核 → 已驳回（rejected，可重新提交）；作废路径：待出库 → 已作废（voided，需填写作废原因，同步作废关联出库单）。状态由 approved 转 partial\_out 的触发条件：首次出库单确认且实际出库量小于订单量；由 partial\_out 转 out\_done 的触发条件：累计出库量达到订单量（允许不超过 out\_weight\_deviation\_rate 配置的偏差率）；由 out\_done 转 completed 的触发条件：收款状态变为 paid。

**出库核销（t\_outbound\_order.status）四态流转：**销售订单审核通过后系统自动创建出库单，初始状态为待出库（pending）。操作员开始执行出库操作时状态转为执行中（processing，库存进入 outbound 过渡态）；操作员确认实际出库重量后状态转为已完成（completed），事务中同步扣减 t\_inventory.current\_stock、写 t\_inventory\_ledger 台账、反写 t\_sales\_order 的 actual\_out\_weight 和 out\_status；作废路径：执行中 → 已作废（voided），需同步释放库存 outbound 过渡态。一个销售订单允许关联多条出库单（分批出库场景），每条出库单独立流转。

**收款记录（t\_sales\_payment.status）三态流转：**出库单全部完成后系统自动创建收款单，初始状态未收款（unpaid）。每次收款操作后累计已收金额，若累计金额小于应收总额则状态为部分收款（partial）；达到应收总额时状态自动变为已收款（paid），同步反写 t\_sales\_order.payment\_status = paid，触发订单状态转为已完成（completed）。部分出库场景下应收金额按实际出库量动态计算（actual\_out\_weight × actual\_sale\_price），不以订单总额为准，避免未出库部分被提前计入应收。

**对账单（t\_reconciliation.status）五态流转：**操作员手动或系统定期（按月）为指定机构生成对账单，状态为已生成（generated）。对账单发送给下游机构确认后转为待确认（pending）；机构无异议则双方确认，状态为已确认（confirmed），收款状态同步锁定，不允许再新增收款记录；若机构对数据有异议，状态转为有争议（disputed）；争议处理中（resolving）记录争议内容和处理进度，处理完成后重新进入待确认流程。

**库存预扣与实扣机制：**订单审核通过时系统在 t\_inventory 对应记录上标记 reserved\_stock（预扣量）字段，available\_stock = current\_stock − reserved\_stock，后续新订单检查库存充足性时以 available\_stock 为准，防止超卖。出库单确认时将 reserved\_stock 扣减，current\_stock 同步减少，两者同步操作在同一事务中。订单作废时 reserved\_stock 对应释放。

**3.4.4 关键业务规则补充与边界条件**

**价格调整约束：**销售单价不得低于参考销售价格的 85%，不得高于 130%，超出范围需管理员审核并填写说明，阈值通过 t\_system\_param（sale\_price\_min\_ratio / sale\_price\_max\_ratio）配置。审核通过后的订单不允许修改价格，需作废后重建。

**分批出库规则：**一个销售订单允许分 N 次出库（如分批交货），每次创建独立出库单，各出库单的计划出库量之和不得超过订单量。最后一批出库单确认时系统自动校验累计出库量是否满足订单要求，未满足时提示操作员确认是否结束出库（提前结束需填写说明，剩余量标记为未出库）。

**收款偏差容忍：**允许实收金额与应收金额存在不超过 1 元的误差（四舍五入导致），系统自动判定为已收款，避免因浮点精度问题造成收款状态异常。偏差阈值通过 t\_system\_param（payment\_tolerance）配置。

**对账锁定规则：**对账单状态为已确认后，涉及机构的历史订单的收款记录进入锁定状态，不允许新增、修改、作废。若发现错误需要人工干预，必须先将对账单状态回退至有争议，处理完成后重新确认。回退操作需管理员权限，写操作日志。

**作废级联规则：**销售订单作废时：① 若关联出库单状态为待出库，直接作废出库单，释放预扣库存；② 若出库单已执行中，禁止作废订单，提示操作员先作废出库单；③ 若已有收款记录，禁止作废，需先退款并在系统中记录退款流水，再作废。所有级联操作在同一数据库事务中执行。

**逾期应收预警：**销售订单出库完成后超过 payment\_overdue\_days（t\_system\_param，默认 30 天）仍未收款，系统触发逾期预警，在首页展示逾期应收列表，支持一键生成催款提醒（记录催款时间和操作员，写 t\_sales\_collection\_record）。

**3.4.7 进销存三模块联动数据流设计**

销售管理模块是进销存闭环的出口，与收购模块（进）、仓储模块（存）形成完整数据链路。以下是一笔完整销售业务的跨模块数据流转描述，也是系统集成测试的核心用例。

**第一步：创建销售订单（销售模块）。**操作员选择下游机构、粮食品种、储位、计划重量、实际单价，系统实时查询 t\_inventory.available\_stock 验证库存充足性，创建 t\_sales\_order（status=draft）。

**第二步：审核销售订单（销售模块）。**管理员审核通过，事务执行：t\_sales\_order.status → approved；t\_inventory.reserved\_stock += plan\_weight（预扣库存）；自动创建 t\_outbound\_order（status=pending，sales\_order\_id 关联）。

**第三步：执行出库（仓储 + 销售模块联动）。**操作员开始出库，t\_outbound\_order.status → processing，t\_inventory.status → outbound（过渡态）。操作员确认实际出库重量，事务执行：t\_inventory.current\_stock -= actual\_weight；t\_inventory.reserved\_stock -= plan\_weight（清零预扣）；t\_inventory.status → in\_stock；t\_inventory\_ledger 写出库流水；t\_outbound\_order.status → completed；t\_sales\_order.actual\_out\_weight += actual\_weight（反写）；t\_sales\_order.out\_status 更新。

**第四步：自动创建收款单（销售模块）。**检测到 t\_sales\_order.out\_status = 已出库，系统自动创建 t\_sales\_payment（status=unpaid，total\_receivable = actual\_out\_weight × actual\_price）。若分批出库，每批出库完成后更新 total\_receivable。

**第五步：登记收款（销售模块）。**操作员登记收款，写 t\_sales\_payment\_record（流水不可删改）；累计 t\_sales\_payment.received\_amount；判断是否达到应收总额（含偏差容忍）：达到则 t\_sales\_payment.status → paid，同步 t\_sales\_order.payment\_status → paid，t\_sales\_order.status → completed。

**第六步：生成对账单（销售模块，可选）。**操作员按月为机构生成对账单，系统聚合期间所有 completed 订单数据，生成 PDF 对账报告（存 MinIO），双方确认后锁定收款记录，完成完整销售业务闭环。整个链路中，涉及库存数量变动的步骤均写 t\_inventory\_ledger 台账流水，涉及金额变动的步骤均写对应收款流水表，保障全程可审计。

**3.5 硬件数据对接模块**

**3.5.1 核心子功能**

1. **地磅设备管理**

￮ 实现地磅设备的新增、编辑、删除、状态管控（启用/禁用），记录设备编号、设备型号、安装位置、维护时间等信息；

￮ 实现地磅设备与系统的对接，支持设备数据的自动采集、实时同步，将称重数据自动写入称重记录表；

￮ 支持设备故障提醒，当设备长时间无数据上传时，系统自动触发预警；

￮ 记录设备运行日志、维护日志，支持日志的查询、导出，为设备维护提供数据支撑。

2. **仓储设备管理**

￮ 实现温湿度传感器、粮情监测设备等仓储设备的新增、编辑、删除、状态管控；

￮ 实现仓储设备数据的自动采集、实时同步，将温湿度、粮情等数据与库存表关联，按储位展示仓储环境数据；

￮ 配置仓储环境预警规则，当温湿度超出粮食存储要求时，系统自动触发预警（系统内提醒、短信提醒）；

￮ 支持仓储设备数据的统计、分析，生成环境数据趋势图，为粮食仓储安全提供保障。

3. **监控设备管理**

￮ 实现监控设备的新增、编辑、删除、状态管控，记录设备编号、安装位置、监控范围；

￮ 实现监控视频的实时查看、历史回放，视频文件存储于MinIO，支持按时间、位置查询监控视频；

￮ 支持监控视频的截图、保存，为收购、仓储、销售过程中的争议问题提供证据支撑。

4. **硬件数据同步管理**

￮ 实现硬件数据的统一采集、清洗、存储，确保硬件数据与系统数据的一致性；

￮ 支持手动同步硬件数据（针对网络故障、设备离线等场景），实现数据补录；

￮ 记录硬件数据同步日志，包含同步时间、同步数据量、同步结果，同步失败时自动提醒并记录原因。

**3.4.2 业务规则**

1. 硬件设备需先启用，方可进行数据采集和同步；

2. 地磅设备的称重数据自动写入称重记录表后，需经操作员审核方可生效；

3. 仓储设备的环境数据超出预警规则时，系统立即触发预警，同时记录预警时间、预警内容；

4. 监控视频文件按时间、设备编号命名，存储于MinIO，保留周期可配置；

5. 硬件数据同步失败时，需及时提醒操作员，排查问题后重新同步，确保数据无遗漏。

**3.5.2 通信架构设计**

硬件数据对接模块整体分为五层：硬件设备层、协议接入层、数据处理层、数据落地层、实时推送层，各层职责清晰，层间通过标准接口解耦，支持后续新增设备类型时仅扩展协议接入层而不影响上层业务逻辑。

**地磅设备接入（串口/TCP 双模式）：**串口模式（RS-232/RS-485）：服务端使用 jSerialComm 库监听串口，以 9600/19200 波特率读取字节流，按自定义帧格式解析报文（帧头 0xAA 0x55 + 设备ID 2字节 + 数据类型 1字节 + 净重 4字节 + 校验和 1字节 + 帧尾 0xFF）；TCP Socket 模式：基于 Netty 框架实现 TCP Server，监听 9001 端口，采用长连接+心跳包（30秒一次 HB 包）维持连接，地磅数据触发时主动推送 DATA 报文，服务端解析后回复 DATA\_ACK 确认。两种模式共用同一套数据处理管道，通过设备表的 conn\_type 字段（0串口/1TCP）区分路由。

**温湿度传感器 / 粮情设备接入（MQTT）：**使用 EMQX 作为 MQTT Broker，部署于本地服务器，监听 1883 端口（MQTT）和 8883 端口（MQTTS，TLS 加密）。设备连接时使用设备编号作为 ClientId，通过用户名/密码认证（credentials 存于 EMQX 内置数据库）。设备上报 Topic 规范：grain/sensor/{device\_id}/data（温湿度）、grain/grain\_monitor/{device\_id}/data（粮情），QoS 级别设为 1（至少一次送达）。服务端通过 Spring Integration MQTT Adapter 订阅 Topic，消息到达后触发处理管道。消息 Payload 格式为 JSON：{"device\_id":"SN001","temp":24.5,"humidity":65.2,"timestamp":1709001234000}。

**监控摄像头接入（RTSP）：**摄像头通过 RTSP 协议推流，URL 格式：rtsp\://admin:password\@192.168.1.x:554/stream。后端使用 FFmpeg 将 RTSP 流转码为 HLS（m3u8+ts 分片），供前端 Video.js 播放，分片文件临时缓存于本地磁盘（保留最近 30 分钟）。历史录像由 FFmpeg 按小时切片后上传 MinIO，存储路径格式：/hardware/video/{device\_id}/{yyyy-MM-dd}/{HH}.mp4，保留周期通过系统参数 video\_retention\_days 配置（默认 30 天），定时任务每日清理过期文件。

**数据处理管道（HardwareDataService）：**所有设备数据进入统一的处理管道，执行以下步骤：① 合法性校验：设备ID是否存在且状态为在线、数据字段完整性校验、数值范围校验（如净重不得为负数、温度范围 -20～60℃）；② 数据清洗：去除重复数据（以 device\_id + timestamp 为去重键，Redis SET 缓存近 5 分钟数据指纹）、单位换算（设备上报以克为单位则转换为千克）；③ 异常值过滤：温湿度数值超出物理上限则丢弃并记录异常日志；④ 落库：写入对应业务表；⑤ 预警检测：比对预警规则触发告警；⑥ WebSocket 推送：将最新数据推送至前端已订阅的连接。

**实时推送（WebSocket）：**后端基于 Spring WebSocket + STOMP 协议实现服务端推送，前端通过 SockJS 建立连接。推送 Topic 分类：/topic/device/{device\_id}/status（设备状态变化）、/topic/weighbridge/{device\_id}/data（实时称重数据）、/topic/alert/new（新告警通知）、/topic/env/{position\_id}/data（储位环境数据更新）。前端页面按需订阅对应 Topic，关闭页面时自动取消订阅，服务端维护订阅关系，避免无效推送。

**3.5.3 设备状态机设计**

所有硬件设备（地磅、传感器、粮情设备、监控摄像头）统一使用以下五态状态机，状态字段存于 t\_device\_base.status。

**未启用（disabled）：**设备录入系统后的初始状态，不进行数据采集，不参与心跳检测。管理员手动启用后转为在线状态。

**在线（online）：**设备正常工作，心跳包持续接收（串口/TCP 设备）或 MQTT 连接活跃。系统以该设备最后一次心跳时间为基准判断是否超时。Redis 缓存 device:{id}:last\_heartbeat 时间戳，TTL = 90 秒；TTL 自然过期则系统判定为离线。

**离线（offline）：**超过 90 秒未收到心跳包或 MQTT 连接断开，系统自动将设备状态变更为离线，同时写入告警记录（t\_device\_alert），触发短信通知运维人员（离线超过 5 分钟发送）。设备重新上线后自动恢复为在线状态，清除对应告警的未处理标记。

**故障（error）：**连续 3 次接收到校验失败的数据报文，或操作员在界面手动标记为故障。故障状态下立即停止自动采集，触发即时告警，同时记录故障时间和故障描述。故障排除后，操作员在界面确认修复，状态恢复为在线。

**维护中（maintaining）：**操作员主动标记，用于计划性维护场景。维护中设备不触发离线告警，数据采集暂停。维护完成后操作员手动结束维护，状态恢复为在线，系统记录本次维护时长和维护人员。

**3.5.4 数据补录与异常恢复机制**

**地磅断网补录：**地磅设备（TCP 模式）本地缓存网络中断期间的称重数据，网络恢复后批量推送历史数据（最多缓存 1000 条）。服务端接收补录数据时，以 device\_id + weigh\_time 为唯一键进行去重校验，data\_source 字段标记为 2（断网补录），补录数据同样经过完整的校验和审核流程，不绕过业务审核。

**手动补录入口：**操作员可在"硬件数据对接 → 数据补录"页面手动录入称重数据，填写设备ID、称重时间、毛重、皮重，系统自动计算净重，data\_source = 1（手动录入），补录记录写入 t\_weighing\_record 并标注补录原因。温湿度数据同样支持手动批量导入（CSV 格式），用于设备维修期间的历史数据回填。

**数据同步日志：**每次数据采集（自动/手动/补录）均写入 t\_hardware\_sync\_log，记录：sync\_time、device\_id、data\_count（本次同步数据条数）、success\_count、fail\_count、fail\_reason、sync\_type（0实时/1补录/2手动）。操作员可在界面查看近 30 天的同步日志，按设备和状态筛选，快速定位数据丢失原因。

**告警重试机制：**短信告警发送失败时（如运营商超时），系统自动重试 3 次（间隔 30 秒），仍失败则写入告警发送失败日志并在系统首页展示未送达提醒，确保运维人员感知告警。

**3.5.7 预警规则与告警处理规范**

**内置预警规则（系统初始化时预置，可修改）：**小麦仓储：温度 > 30℃ 持续 10 分钟触发重要告警；湿度 > 75% 持续 10 分钟触发重要告警。水稻仓储：温度 > 28℃ 持续 10 分钟触发重要告警；湿度 > 80% 触发告警。粮情设备：粮堆测温点任意一点超过 35℃ 立即触发紧急告警。设备离线：任意设备离线超过 5 分钟触发普通告警，超过 30 分钟升级为重要告警。

**告警去重机制：**同一设备同一告警类型在 recheck\_minutes 间隔内不重复生成新告警记录，而是更新已有告警的最新检测值和告警时间，避免高频刷新导致告警表膨胀。当检测值恢复正常后，系统自动将对应告警标记为已恢复（status = 3），并记录恢复时间，无需操作员手动处理。

**告警处理闭环：**操作员处理告警时必须填写处理结果（如"已调整仓库通风设备"、"已联系维修人员"），处理结果与告警记录永久绑定。管理员可在统计分析模块查看告警处理率、平均处理时长、高频告警设备排名，用于指导设备维护计划制定。

**硬件对接模块整体业务规则补充：**地磅设备的称重数据必须经操作员审核后方可关联收购单据，自动采集不等于自动生效，审核环节为必要人工确认节点。同一储位的温湿度传感器最多绑定一个（t\_device\_base 和 t\_storage\_position 一对一约束）；粮情监测设备按粮堆布设，允许一个储位对应多个粮情设备。监控视频历史录像的查询和下载操作记录在 t\_operation\_log 中，关联操作人、查询时间范围和摄像头ID，满足安防合规要求。

**3.6 统计分析模块**

**3.6.1 核心子功能**

1. **收购统计分析**

￮ 支持按时间区间（日/周/月/季/年）、粮食种类、农户、操作用户统计收购数据，包含收购数量、收购金额、农户售粮户数；

￮ 生成收购数据趋势图、饼图（粮食种类占比）、柱状图（每日/每月收购量），实现数据可视化展示；

￮ 支持收购统计报表的打印、导出，支持自定义报表字段；

￮ 实现收购数据的对比分析（如本年与上年同期对比、本月与上月对比），为收购计划制定提供数据支撑。

2. **仓储统计分析**

￮ 支持按时间区间、粮食种类、储位统计仓储数据，包含库存总量、入库总量、出库总量、盘点差异率；

￮ 生成库存状态可视化图表，展示各类粮食库存占比、储位使用情况；

￮ 统计仓储预警信息，包含预警类型、预警次数、处理结果，分析预警原因；

￮ 生成仓储统计报表，支持打印、导出，为库存管控提供数据支撑。

3. **销售统计分析**

￮ 支持按时间区间、粮食种类、下游机构、操作用户统计销售数据，包含销售数量、销售金额、机构采购户数；

￮ 生成销售数据趋势图、饼图（下游机构采购占比）、柱状图（每日/每月销售量），实现数据可视化展示；

￮ 支持销售数据的对比分析（如本年与上年同期对比、本月与上月对比），分析销售趋势；

￮ 生成销售统计报表，支持打印、导出，为销售计划制定提供数据支撑。

4. **经营数据汇总**

￮ 实现代收点经营数据的一键汇总，包含总收购量、总库存量、总销售量、总收购金额、总销售金额、利润（粗略计算）；

￮ 生成经营数据仪表盘，在系统首页醒目展示核心经营指标，实现经营状态的实时监控；

￮ 支持按时间区间生成经营汇总报表，包含各类明细数据，为代收点经营决策提供全面的数据支撑；

￮ 支持经营数据的备份、归档，实现历史经营数据的长期存储和查询。

**3.6.2 业务规则**

1. 统计分析数据均来源于系统各模块的业务数据，实时同步，确保数据准确性；

2. 支持自定义统计条件、报表字段，满足不同的统计分析需求；

3. 所有统计报表、可视化图表均支持打印、导出（Excel/PDF/图片）；

4. 经营数据汇总的利润计算为粗略计算（销售金额-收购金额），不含仓储、人工等成本；

5. 统计分析数据支持按用户权限展示，管理员可查看全量数据，操作员仅可查看所属业务范围的数据。

**3.6.3 计算架构与性能设计**

**实时聚合查询（在线报表）：**日报、周报等小数据量报表（结果集 < 1 万行）采用实时 SQL 聚合查询，结合 MySQL 索引覆盖和分页优化，响应时间控制在 2 秒以内。聚合结果在 Redis 中缓存 5 分钟（key 格式：stats:{report\_type}:{dimension\_hash}:{date\_range\_hash}），缓存命中率预期达 80% 以上，有效降低数据库查询压力。业务数据写入时通过 Spring Event 异步发布数据变更事件，监听器收到事件后主动删除受影响的统计缓存（Cache-Aside 策略），确保数据一致性。

**夜间预计算（离线报表）：**月度、年度、同比对比等大范围报表由 Quartz 定时任务在每日凌晨 2:00 预计算，结果写入 t\_stats\_snapshot 快照表，前端查询时直接读取快照，不触发实时聚合。快照表按报表类型和统计周期建索引，查询响应 < 200 毫秒。若当日有数据补录或订正操作，任务在次日凌晨重新计算前一天快照（通过设置 is\_valid = 0 标记过期，重算后更新）。

**大报表异步生成：**单次导出超过 10 万行数据的报表（如全年收购明细）采用异步任务模式：前端请求后立即返回任务ID，后端在独立线程池中生成报表文件（EasyExcel 流式写入 Excel），完成后上传至 MinIO 并通过 WebSocket 推送下载通知（含预签名 URL，有效期 2 小时）。任务进度可在"我的报表"页面查看，超过 30 分钟未完成自动标记失败，支持重新触发。

**数据权限隔离：**StatisticsService 统一注入当前登录用户的权限上下文（data\_scope 字段），管理员查询不附加过滤条件；操作员查询自动附加 WHERE create\_user\_id = {current\_user\_id} 或 WHERE operator\_id = {current\_user\_id} 条件，确保操作员只看到本人经手的业务数据。数据权限在 Service 层实现（非 SQL 层），避免绕过权限的直接 SQL 查询。

**3.6.4 核心指标定义与计算规则**

**收购维度核心指标：**收购总量（kg）= SUM(t\_purchase\_order.actual\_weight) WHERE status IN (approved, completed)；收购总金额（元）= SUM(t\_purchase\_order.total\_amount)；农户参与数 = COUNT(DISTINCT farmer\_id)；平均收购单价 = 收购总金额 / 收购总量；收购完成率 = 已完成单据数 / 总单据数 × 100%；付款完成率 = 已付款单据数 / 总单据数 × 100%。品种占比 = 各品种收购量 / 总收购量，饼图展示。

**仓储维度核心指标：**当前库存总量 = SUM(t\_inventory.current\_stock) WHERE status = in\_stock；储位使用率 = 已使用储位数 / 总启用储位数 × 100%；库存周转率（月）= 当月出库量 / 月均在库量；盘点差异率 = ABS(SUM(diff\_qty)) / SUM(book\_qty) × 100%（按次盘点统计）；台账平衡率 = 实时校验库存守恒公式通过次数 / 总校验次数 × 100%。

**销售维度核心指标：**销售总量（kg）= SUM(t\_sales\_order.actual\_weight) WHERE status IN (completed, partial\_out)；销售总金额（元）= SUM(t\_sales\_order.total\_amount)；应收款总额 = SUM(total\_amount) − SUM(received\_amount)；收款率 = 已收款金额 / 销售总金额 × 100%；客户活跃数 = COUNT(DISTINCT org\_id)。

**经营汇总核心指标：**毛利润（粗略）= 销售总金额 − 收购总金额（不含仓储成本、人工成本，仅供参考，系统界面明确标注"粗略估算"）；进销存平衡验证：收购总入库量 ≈ 当前库存总量 + 销售出库总量 + 盘点损耗总量，三者之差应在允许误差范围（0.1%）以内；告警处理率 = 已处理告警数 / 总告警数 × 100%；设备在线率 = 当日在线设备数 / 启用设备总数 × 100%。

**同比 / 环比计算：**同比 = (本期值 − 上年同期值) / 上年同期值 × 100%，上年同期值从 t\_stats\_snapshot 快照读取；环比 = (本期值 − 上期值) / 上期值 × 100%，上期值为上一自然月/周/日的实时或快照数据。同比/环比结果缓存于 Redis（TTL 10 分钟），前端以箭头 + 颜色（红涨绿降/红降绿涨，根据指标语义定义）展示趋势。

**四、数据设计**

**4.1 数据库设计**

**4.1.1 数据库选型与规范**

采用MySQL 8.0关系型数据库，遵循系统统一的命名规范：

• 数据库名：grain\_purchase\_inventory\_system；

• 表名：t\_+模块名称，小写字母+下划线，如t\_user、t\_grain、t\_inventory；

• 字段名：小写字母+下划线，语义清晰，避免使用关键字，如user\_id、grain\_type、inventory\_num；

• 索引名：idx\_+表名+字段名，如idx\_user\_phone、idx\_grain\_type；

• 主键：所有表均设置主键，优先使用自增INT类型，提升查询效率。

**4.1.2 核心表结构**

系统数据库表分为五大类：用户相关表、业务交易相关表、库存相关表、硬件数据相关表、基础配置相关表，核心表已在数据库设计文档中定义，包含t\_user（用户表）、t\_role（角色表）、t\_farmer（农户表）、t\_downstream\_org（下游机构表）、t\_grain（粮食表）、t\_purchase\_reserve（收购预约表）、t\_weighing\_record（称重记录表）、t\_purchase\_order（收购单据表）、t\_inventory（库存表）、t\_inventory\_check（库存盘点表）、t\_sales\_order（销售订单表）、t\_weighbridge（地磅设备表）等，各表通过外键关联，确保数据一致性。

**4.1.3 索引设计**

为提升数据库查询效率，对各表的常用查询字段、外键字段建立索引，核心索引设计如下：

• t\_user：idx\_user\_phone（phone）、idx\_user\_role\_id（role\_id）、idx\_user\_status（status）；

• t\_farmer：idx\_farmer\_user\_id（user\_id）、idx\_farmer\_id\_card（id\_card）；

• t\_downstream\_org：idx\_org\_user\_id（user\_id）、idx\_org\_audit\_status（audit\_status）；

• t\_grain：idx\_grain\_type（grain\_type）、idx\_grain\_grade（grain\_grade）；

• t\_purchase\_reserve：idx\_reserve\_farmer\_id（farmer\_id）、idx\_reserve\_create\_time（create\_time）；

• t\_weighing\_record：idx\_weigh\_record\_weighbridge\_id（weighbridge\_id）、idx\_weigh\_record\_create\_time（create\_time）；

• t\_purchase\_order：idx\_po\_farmer\_id（farmer\_id）、idx\_po\_grain\_id（grain\_id）、idx\_po\_create\_time（create\_time）；

• t\_inventory：idx\_inventory\_grain\_id（grain\_id）、idx\_inventory\_storage\_position（storage\_position）、idx\_inventory\_quality\_status（quality\_status）；

• t\_inventory\_check：idx\_check\_inventory\_id（inventory\_id）、idx\_check\_create\_time（create\_time）；

• t\_sales\_order：idx\_so\_org\_id（org\_id）、idx\_so\_grain\_id（grain\_id）、idx\_so\_order\_status（order\_status）。

**4.2 非结构化数据存储**

**4.2.1 存储选型**

采用MinIO分布式文件存储，用于存储系统中的非结构化数据，包括监控视频、业务单据（收购、入库、出库、销售）、资质文件、盘点照片、硬件运行日志等。

**4.2.2 存储规范**

1. **文件命名**：模块名\_文件类型\_时间戳\_唯一ID，如purchase\_order\_20260227\_001.pdf、inventory\_check\_photo\_20260227\_002.jpg；

2. **文件路径**：按模块划分存储目录，如/minio/grain/purchase/（收购相关文件）、/minio/grain/inventory/（仓储相关文件）、/minio/grain/hardware/（硬件相关文件）；

3. **关联方式**：在对应的数据表中增加文件路径字段（VARCHAR(255)），存储MinIO中的文件访问路径，实现结构化数据与非结构化数据的关联；

4. **文件权限**：按用户角色设置文件访问权限，仅授权用户可查看、下载对应文件，敏感文件加密存储。

**4.3 数据交互与同步**

1. **模块间数据交互**：通过数据库表外键关联实现，如收购单据表关联称重记录表、农户表，销售订单表关联下游机构表、库存表，确保数据联动，减少重复录入；

2. **硬件与系统数据同步**：硬件设备通过接口将数据实时推送至系统，系统接收后自动清洗、存储至对应数据表，支持手动补录同步，确保数据无遗漏；

3. **库存数据同步**：入库、出库、盘点差异处理操作完成后，系统实时更新库存表，确保库存数据与实际业务数据一致；

4. **数据事务管理**：所有涉及多表操作的业务（如收购单据生成、出库单确认）均采用事务管理，任一操作失败则事务回滚，保障数据一致性。

**五、界面设计**

**5.1 整体界面布局**

系统采用**左侧功能菜单+顶部导航栏+右侧主操作区**的经典布局，适配电脑端主流分辨率，界面元素大小适中，操作路径清晰，降低基层操作人员的学习成本：

3. **右侧主操作区**：展示具体的操作界面、数据列表、图表、表单等内容，为核心操作区域，支持多标签页操作，方便用户切换业务。

1. **简洁性**：剔除冗余的界面元素，采用简约的设计风格，色彩搭配以浅色系为主，重点信息（预警、待办）用醒目颜色标注；

2. **一致性**：统一界面组件样式（按钮、输入框、表格、弹窗）、操作方式（查询、新增、编辑、删除），确保各模块界面风格一致；

3. **易用性**：表单设计尽可能减少人工录入，支持核心数据自动回填、下拉选择、批量导入，按钮布局清晰，操作指令明确；

4. **可视化**：核心数据（经营指标、库存状态、收购/销售趋势）采用图表（饼图、柱状图、折线图、仪表盘）展示，直观易懂；

5. **反馈性**：所有操作（新增、编辑、删除、确认）完成后，系统给出明确的成功/失败提示，失败提示说明具体原因，便于用户排查问题；

6. **适配性**：支持主流浏览器（Chrome、Edge、Firefox），界面自适应不同屏幕分辨率，无复杂的交互效果，适配乡镇基层操作人员的使用习惯。

1. **系统首页**：展示核心经营数据仪表盘、待办事项（待审核单据、待付款、待收款、待盘点）、预警信息、业务数据趋势图，实现经营状态的一站式监控；

2. **数据列表界面**：采用分页展示，支持多条件查询、高级搜索、批量操作、数据导出、打印，列表字段可自定义显示/隐藏；

3. **表单操作界面**：采用分步表单（针对复杂业务，如收购单据、销售订单），表单字段按业务逻辑排序，必填字段标注星号，支持表单暂存、重置；

4. **图表展示界面**：支持图表类型切换（饼图、柱状图、折线图）、数据筛选、图表导出（图片/PDF），支持查看图表明细数据；

5. **预警提醒界面**：展示预警类型、预警内容、预警时间、处理状态，支持一键处理、标记已读，未处理预警在系统首页醒目展示。

**六、性能与安全设计**

**6.1 性能设计**

1. **查询优化**：对数据库常用查询字段、外键字段建立索引，减少数据库全表扫描，提升查询效率；对高频查询的静态数据（粮食种类、角色权限）进行本地缓存，减少数据库交互次数；

2. **数据分页**：所有数据列表均采用分页展示，默认每页20条数据，支持自定义分页大小，避免大数据量加载导致界面卡顿；

3. **批量操作优化**：对批量新增、批量导入、批量审核等操作进行SQL优化，采用批量插入/更新语句，减少数据库访问次数；

4. **前端优化**：对前端页面进行懒加载，仅加载当前视图的内容；压缩前端静态资源（JS、CSS、图片），提升页面加载速度；

5. **硬件数据采集优化**：硬件数据采用增量采集、异步同步方式，避免大量数据同时上传导致系统压力过大；

6. **报表生成优化**：统计分析报表采用异步生成方式，生成完成后通知用户，避免长时间等待；对历史报表数据进行缓存，支持快速查询。

**6.2 安全设计**

1. **身份认证**：采用账号密码登录方式，密码加密存储，支持密码复杂度校验（数字+字母+特殊字符）；登录失败多次后锁定账号，防止暴力破解；支持登录日志查询，记录登录时间、登录IP、登录状态；

2. **权限控制**：基于RBAC权限模型，为不同角色分配不同的菜单权限、按钮权限、数据权限，遵循最小权限原则；用户仅可访问所属权限范围内的功能和数据，杜绝越权操作；

3. **数据加密**：对敏感数据（密码、身份证号、银行卡号、收购/销售价格）进行加密存储，不可明文展示；传输过程中采用HTTPS协议，保障数据传输安全；

4. **操作留痕**：所有业务操作（新增、编辑、删除、审核、付款、收款）均记录操作日志，包含操作用户、操作时间、操作内容、操作前/后数据变化，日志永久保存，不可删除，实现操作可追溯；

5. **数据防篡改**：核心业务单据（收购、入库、出库、销售）审核通过后不可随意修改，仅可作废后重新生成；库存数据仅可通过入库、出库、盘点差异处理操作修改，杜绝人工随意篡改；

6. **数据备份与恢复**：配置数据库定期自动备份策略（每日备份），备份文件存储于异地，防止数据丢失；支持按备份时间点恢复数据，应对数据库故障、数据误删等场景；

7. **文件安全**：MinIO中的文件设置访问权限，仅授权用户可查看、下载；文件上传时进行病毒扫描，防止恶意文件上传；对重要文件（资质文件、业务单据）进行加密存储；

8. **系统安全**：服务器开启防火墙，仅开放必要的端口；定期更新服务器系统和应用程序补丁，修复安全漏洞；禁止非授权人员访问服务器。

**七、系统扩展性设计**

为适配乡镇粮食代收点后续业务发展和智能化升级需求，系统采用模块化设计，预留丰富的拓展接口和功能拓展空间，具体扩展性设计如下：

1. **功能模块拓展**：预留**粮食调拨模块、仓储成本核算模块、客户管理模块、供应链管理模块**等接口，可根据业务需求后续开发，模块间松耦合，不影响现有系统运行；

2. **硬件对接拓展**：预留**智能粮情监测设备、自动称重设备、无人值守地磅、库存扫码设备**等硬件对接接口，支持后续智能化硬件接入，实现无人化、自动化操作；

3. **终端适配拓展**：预留**移动端API接口**，支持后续开发手机端、平板端应用，实现移动化操作（如农户线上预约、操作员移动盘点、管理人员远程监控）；

4. **数据对接拓展**：预留**上级粮食管理部门、农业农村部门、税务部门**等政府平台的数据对接接口，支持数据上报、信息共享，满足政策监管要求；

5. **性能拓展**：系统部署架构支持从单机部署升级为集群部署，当业务量增大时，可增加应用服务器、数据库服务器，提升系统处理能力；

6. **个性化配置拓展**：预留**自定义表单、自定义报表、自定义预警规则**等功能，支持代收点根据自身经营需求灵活配置系统功能，提升系统适配性。

**八、系统运维与维护设计**

**8.1 系统运维设计**

1. **运维界面**：开发专门的系统运维界面，仅管理员可访问，包含服务器状态监控、数据库状态监控、MinIO存储状态监控、硬件设备运行状态监控，实现系统运行状态的一站式监控；

2. **故障预警**：配置系统故障预警规则，当服务器、数据库、硬件设备出现异常时（如服务器宕机、数据库连接失败、设备离线），系统自动触发预警（短信+系统内提醒），通知运维人员；

3. **日志管理**：实现系统运行日志、数据库操作日志、硬件数据同步日志的统一管理，支持日志查询、筛选、导出、归档，为故障排查提供数据支撑；

4. **远程运维**：预留远程运维接口，支持运维人员远程登录系统进行故障排查、系统升级，减少现场运维成本。

**8.2 系统维护设计**

1. **版本升级**：系统采用模块化设计，支持版本的增量升级，升级过程中不影响现有业务的正常运行，升级完成后提供升级日志，说明升级内容和注意事项；

2. **数据维护**：支持数据库数据的批量导入/导出、数据清洗、数据修复，应对数据异常、数据迁移等场景；支持MinIO文件的批量备份、归档、删除，释放存储空间；

3. **硬件维护**：系统记录硬件设备的维护时间、维护内容、维护人员，支持维护提醒，当设备达到维护周期时，系统自动提醒运维人员进行设备维护；

4. **技术支持**：配备完善的系统操作手册、设计文档、故障排查手册，为基层操作人员和运维人员提供技术支持；预留技术支持入口，方便用户提交问题、获取帮助。

**九、总结**

本乡镇粮食代收点进销存系统基于乡镇粮食代收点的实际业务需求设计，采用B/S架构、分层设计思想和模块化开发模式，实现了基础管理、收购管理、仓储管理、销售管理、硬件数据对接、统计分析六大核心功能的一体化设计，覆盖粮食进销存全业务流程。

系统在设计过程中充分考虑了乡镇基层操作人员的使用习惯，突出操作便捷性、数据精准性、管理可视化；同时建立了完善的性能与安全设计体系，保障系统的稳定运行和数据安全；预留了丰富的拓展接口和功能空间，适配后续业务升级和智能化改造需求。

本系统的落地将有效解决传统乡镇粮食代收点人工管理的痛点，实现业务的数字化、标准化、高效化管理，提升代收点的经营管理效率和决策水平，为乡镇粮食产业的数字化发展提供支撑。
