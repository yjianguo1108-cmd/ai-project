# 乡镇粮食代收点进销存系统概要设计文档

&#x20;

## 一、系统概述

### 1.1 系统定位

本系统是面向乡镇粮食代收点的一体化进销存管理系统，聚焦粮食收购、仓储、销售全业务流程，整合用户管理、预约收购、称重计量、库存管控、订单销售、硬件数据对接等核心功能，实现乡镇粮食代收点业务的数字化、标准化、高效化管理，解决传统代收点人工记账效率低、数据易出错、库存管控不精准、业务追溯难等问题，适配基层操作人员的使用习惯，同时为代收点经营决策提供数据支撑。

### 1.2 设计目标

1. 业务全覆盖：覆盖乡镇粮食代收点从农户预约、粮食收购、称重、入库、仓储盘点到下游销售、出库的全业务链路，实现进销存一体化管理；
2. 数据精准化：建立统一的数据管理体系，实现各环节数据实时同步、自动流转，杜绝人工录入误差，保障账实相符、数据一致；
3. 操作便捷化：简化操作流程，减少重复录入，适配乡镇基层操作人员的计算机操作水平，降低系统学习和使用成本；
4. 管理可视化：通过数据统计、图表展示，实现库存状态、交易数据、业务进度的实时监控，为管理人员提供直观的经营视角；
5. 数据安全性：建立完善的权限控制、操作留痕、数据加密机制，保障用户信息、交易数据、库存数据的安全，满足数据追溯要求；
6. 系统可扩展：采用模块化设计，预留硬件对接、功能拓展接口，适配代收点后续业务升级和智能化改造需求。

### 1.3 设计范围

本系统覆盖基础管理模块、收购管理模块、仓储管理模块、销售管理模块、硬件数据对接模块、统计分析模块六大核心模块，同时包含数据库设计、系统架构设计、界面设计、性能与安全设计等内容，实现从用户身份认证、业务操作到数据统计、系统维护的全流程设计，适配乡镇粮食代收点的日常经营管理需求。

### 1.4 设计原则

1. 实用性原则：以乡镇粮食代收点实际业务需求为核心，剔除冗余功能，确保系统功能贴合实际操作场景；
2. 模块化原则：按业务功能划分子模块，模块间松耦合、内聚性强，便于系统开发、维护和功能拓展；
3. 数据一致性原则：建立统一的数据标准，实现各模块数据自动同步、联动更新，确保收购、仓储、销售数据高度一致；
4. 易用性原则：采用简洁的界面布局、清晰的操作流程，支持一键查询、批量操作、数据导入导出，提升操作效率；
5. 安全性原则：遵循最小权限原则，建立角色化权限控制体系，所有操作全程留痕，敏感数据加密存储；
6. 兼容性原则：适配主流计算机操作系统和浏览器，支持与地磅、温湿度传感器等基层常用硬件设备对接；
7. 可维护性原则：系统代码、数据库设计遵循统一规范，配备完善的设计文档和操作手册，便于后期维护和升级。

## 二、系统整体架构设计

### 2.1 技术架构

系统采用 B/S（浏览器 / 服务器）架构，结合分层设计思想，分为表现层、业务逻辑层、数据访问层、数据层四层架构，各层级职责清晰、解耦设计，便于开发、维护和扩展，整体技术架构如下：

1. 表现层：为用户提供可视化操作界面，基于主流前端技术开发，支持电脑端主流浏览器（Chrome、Edge、Firefox），实现界面展示、用户交互、操作指令传递等功能，适配乡镇基层操作人员的使用场景；
2. 业务逻辑层：系统核心层，包含六大核心模块的业务规则实现、流程控制、跨模块数据联动、权限校验等功能，接收表现层的操作指令，处理后传递至数据访问层，同时将处理结果反馈至表现层；
3. 数据访问层：负责与数据库和文件存储的交互，封装数据 CRUD 操作，遵循统一的数据库访问规范，实现结构化数据与 MySQL 数据库的交互、非结构化数据与 MinIO 文件存储的对接，为业务逻辑层提供数据支撑；
4. 数据层：包含 MySQL 8.0 关系型数据库和 MinIO 分布式文件存储，其中 MySQL 用于存储用户信息、交易数据、库存数据等结构化数据，MinIO 用于存储监控视频、业务单据、资质文件、盘点照片等非结构化数据，保障数据存储的规范和高效。

#### 2.1.1 具体技术选型

- 前端技术栈：Vue 3 + Vite + Element Plus + Pinia + Axios + Vue Router 4
- 后端技术栈：Spring Boot 3.2.x + Java 17 + MyBatis-Plus
- 缓存方案：Redis 7.x
- 身份认证与安全：JWT、BCrypt、AES-256-GCM、HTTPS
- 硬件通信协议：地磅（RS-232/RS-485、TCP Socket/Netty）、温湿度 / MQTT（EMQX）、监控（RTSP→HLS）
- 文件存储：MinIO
- 构建与部署：Nginx、JAR、单机一体化部署

### 2.2 系统部署架构

系统采用单机部署模式，部署于乡镇代收点本地服务器，包含应用服务器、数据库服务器、文件存储服务器，三者一体化部署。

### 2.3 模块整体划分

1. 基础管理模块
2. 收购管理模块
3. 仓储管理模块
4. 销售管理模块
5. 硬件数据对接模块
6. 统计分析模块

## 三、核心模块详细设计

### 3.1 基础管理模块

#### 3.1.1 核心子功能

1. 用户管理
2. 角色权限管理
3. 农户档案管理
4. 下游机构档案管理
5. 基础配置管理

#### 3.1.2 业务规则

- 用户名、手机号、农户身份证号唯一
- 敏感信息加密存储
- 禁用账号无法登录
- 基础数据被引用后不可删除，仅可禁用
- 所有操作记录日志

#### 3.1.3 状态流转设计

- 用户账号状态：正常 → 禁用 / 锁定
- 下游机构审核状态：待审核 → 审核通过 / 驳回
- 基础数据状态：启用 → 禁用

#### 3.1.4 核心接口设计（RESTful API）

- 用户管理：/api/v1/system/user
- 角色权限：/api/v1/system/role
- 农户档案：/api/v1/farmer/profile
- 下游机构：/api/v1/downstream/org
- 基础配置：/api/v1/config

#### 3.1.5 关键数据表字段设计

t\_user、t\_role、t\_permission、t\_user\_role、t\_role\_permission、t\_farmer、t\_downstream\_org、t\_grain、t\_storage\_position、t\_system\_param

***

### 3.2 收购管理模块

#### 3.2.1 核心子功能

1. 收购预约管理
2. 称重计量管理
3. 收购单据管理
4. 收购付款管理

#### 3.2.2 业务规则

- 预约→称重→单据→付款逐级审核
- 审核通过后自动触发入库与付款单
- 价格、重量偏差受控
- 事务保证一致性

#### 3.2.3 收购全流程状态机设计

- 预约单：待审核 → 已审核 → 已排期 / 已取消
- 称重记录：待审核 → 已确认 / 已作废
- 收购单据：草稿 → 待审核 → 审核通过 → 已完成 / 已作废
- 付款单：未付款 → 部分付款 → 已付款

#### 3.2.4 核心接口设计

/api/v1/purchase/reserve、/api/v1/purchase/weighing、/api/v1/purchase/order、/api/v1/purchase/payment

#### 3.2.5 关键数据表

t\_purchase\_reserve、t\_weighing\_record、t\_purchase\_order、t\_purchase\_payment

***

### 3.3 仓储管理模块

#### 3.3.1 核心子功能

1. 入库管理
2. 库存管理
3. 盘点管理
4. 出库管理

#### 3.3.2 业务规则

- 入库需依据、出库校验库存
- 盘点期间锁定库存
- 差异处理完成前禁止出库
- 库存实时更新

#### 3.3.3 库存记录状态机设计

待入库 → 在库 → 盘点锁定 → 差异待处理 → 出库中

#### 3.3.4 盘点闭环流程设计

创建计划 → 发布锁定 → 录入实盘 → 审核 → 差异处理 → 完成归档

#### 3.3.5 库存预警机制

库存不足、积压、超期预警

#### 3.3.6 核心接口

/api/v1/inventory/inbound、/api/v1/inventory/stock、/api/v1/inventory/check、/api/v1/inventory/outbound

#### 3.3.7 关键数据表

t\_inventory、t\_inventory\_ledger、t\_inbound\_order、t\_outbound\_order、t\_inventory\_check、t\_inventory\_check\_detail、t\_inventory\_adjustment

***

### 3.4 销售管理模块

#### 3.4.1 核心子功能

1. 销售订单管理
2. 出库核销管理
3. 销售收款管理
4. 客户对账管理

#### 3.4.2 业务规则

- 仅审核通过机构可下单
- 库存预扣防止超卖
- 价格、出库偏差受控
- 对账锁定历史数据

#### 3.4.3 三段状态联动设计

销售订单 → 出库核销 → 收款记录 联动流转

#### 3.4.4 核心接口

/api/v1/sales/order、/api/v1/sales/outbound、/api/v1/sales/payment、/api/v1/sales/reconciliation

#### 3.4.5 关键数据表

t\_sales\_order、t\_sales\_outbound、t\_sales\_payment、t\_sales\_payment\_record、t\_reconciliation、t\_sales\_collection\_record

***

### 3.5 硬件数据对接模块

#### 3.5.1 核心子功能

1. 地磅设备管理
2. 仓储设备管理
3. 监控设备管理
4. 硬件数据同步管理

#### 3.5.2 通信架构设计

- 地磅：串口、TCP Socket（Netty）
- 温湿度 / 粮情：MQTT（EMQX）
- 监控：RTSP→HLS
- 实时推送：WebSocket

#### 3.5.3 设备状态机设计

未启用 → 在线 → 离线 → 故障 → 维护中

#### 3.5.4 数据补录与异常恢复机制

断网补录、手动补录、同步日志、告警重试

#### 3.5.5 核心接口

/api/v1/hardware/device、/api/v1/hardware/weighing、/api/v1/hardware/env、/api/v1/hardware/alert、/api/v1/hardware/camera

#### 3.5.6 关键数据表

t\_device\_base、t\_env\_record、t\_device\_alert、t\_alert\_rule、t\_hardware\_sync\_log、t\_device\_maintain\_record

***

### 3.6 统计分析模块

#### 3.6.1 核心子功能

1. 收购统计分析
2. 仓储统计分析
3. 销售统计分析
4. 经营数据汇总

#### 3.6.2 计算架构与性能设计

实时聚合 + Redis 缓存、凌晨预计算、异步大报表、数据权限隔离

#### 3.6.3 核心指标定义

收购总量 / 金额、库存总量、销售总量 / 金额、毛利润、周转率、差异率、收款率

#### 3.6.4 核心接口

/api/v1/stats/dashboard、/api/v1/stats/purchase、/api/v1/stats/inventory、/api/v1/stats/sales、/api/v1/stats/business

#### 3.6.5 关键数据表

t\_stats\_snapshot、t\_report\_task
